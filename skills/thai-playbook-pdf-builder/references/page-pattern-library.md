# Page Pattern Library

Use these patterns to design A4 Thai Simple Action Playbook pages quickly. Combine patterns, but keep one main idea per page.

## Cover

Purpose: make the value obvious in 5 seconds and feel connected to the rest of the PDF.

Include:

- Title focused on the output, not a vague topic.
- One-sentence promise.
- Simple 3-5 step method preview.
- Source credit short line if source-based.
- Same cream/ivory visual palette as the inside pages.

Avoid:

- Long intro paragraphs.
- Dark or black cover that feels separate from the PDF.
- Decorative visuals that do not explain the topic.

## Problem This Playbook Solves

Use a simple before/after frame:

| Reader Pain | What This Playbook Helps With |
|---|---|
| [confusion/manual work/unclear process] | [specific output or workflow] |

## What You Can Do After Reading

Use this near the start:

| Output | What The Reader Gets |
|---|---|
| Main output | [specific asset/workflow/result] |
| Reusable asset | [template/checklist/prompt/SOP] |
| Quality check | [how they know it is ready] |
| Next use | [where to apply it after reading] |

## Start Here

Use a path selector:

| Situation | Time | Start With | Result |
|---|---:|---|---|
| Need immediate result | 15 min | Quick Win | small usable output |
| New to the topic | 20 min | Hard Words + Simplest Example | enough context to follow |
| Real project | 45-90 min | Step 1 | work-ready output |
| Team/SOP | 1-2 hours | Decision Guide + Cheat Sheet | repeatable process |

## Hard Words In Plain Thai

Use before advanced instructions:

| Term | Plain Thai Meaning | Tiny Example |
|---|---|---|
| [term] | [meaning] | [example] |

Keep each explanation short enough for a beginner to understand in one pass.

## Simplest Example First

Use before the full method:

```text
Situation:
[plain work scenario]

Input:
[small realistic input]

Do:
[1-3 simple actions]

Output:
[small result]

Why this matters:
[bridge to the full method]
```

## What To Prepare Before Starting

| Need | Example | If Missing |
|---|---|---|
| Input | transcript, raw notes, data, brief | use the sample input |
| Access/tool | app, account, file | use manual fallback |
| Decision | audience, goal, format | choose default path |

## Quick Win

Use this pattern:

```text
Goal:
[one small result]

Do this:
1. [step]
2. [step]
3. [step]

Copy this:
[prompt/template]

Check:
[1-3 criteria]
```

## Simple Method Overview

Use 3-5 compact cards:

```text
1. [verb + short name]
What the reader does.
Output from this step.
```

Each method step must become one or more step pages later.

## Step Page

Recommended block order:

1. Step goal.
2. Use when.
3. Inputs needed.
4. Steps.
5. Example input/output.
6. Prompt, checklist, or worksheet.
7. Checkpoint.
8. Common mistake and fix.
9. 15-minute action.

Make the page useful even if the reader skips the previous page.

## Worksheet

Use fillable labels:

```text
Goal:
Input I have:
Audience:
Constraint:
First draft:
What good looks like:
Next action:
```

Keep worksheets short enough to complete in one sitting.

## Prompt Box

Every prompt box should include:

- When to use.
- Copy-ready prompt.
- Variables to replace.
- How to judge output.

Avoid prompts that depend on hidden context.

## Full Worked Example

Use this sequence:

| Step | Content |
|---|---|
| Situation | real reader scenario |
| Input | raw material or constraints |
| Process | method steps applied |
| Draft output | first result |
| Problem | weak point, risk, or missing source |
| Fix | exact improvement |
| Final output | usable result |
| Quality check | pass/fix result |

## Copy Box

Use short copy-ready blocks:

```text
Use when:
Copy this:
[template]
Replace:
[variables]
Check before use:
[criteria]
```

## Fix Guide

| Problem | Likely Cause | Fix |
|---|---|---|
| [too vague] | input is unclear | add goal, audience, and output format |
| [wrong detail] | source was not checked | return to source evidence |
| [not usable] | missing next step | add checklist or decision rule |

## Good vs Weak Output

Use paired cards or a 4-row table:

| Part | Content |
|---|---|
| Weak Output | [bad example] |
| Why It Fails | [1-3 reasons] |
| Good Output | [better example] |
| Fix | [steps] |

## Decision Guide

Use when readers must choose a path:

| If You Have | Choose | Why |
|---|---|---|
| little time | Quick Win | gets usable output fast |
| real project | Step-by-step | enough examples and QA |
| team/process need | SOP path | turns into workflow/SOP |
| strong context already | Copy Box | fastest reuse |

## Cheat Sheet

Use a compact final reference:

| Do | Avoid | Check |
|---|---|---|
| [best action] | [common mistake] | [quality criterion] |

## Quality Check

Use pass/fix with a clear correction:

| Check | Pass/Fix | Fix If Needed |
|---|---|---|
| Big Idea is clear | pass/fix | rewrite promise and method |
| Output is usable | pass/fix | add steps, examples, and templates |
| Source trust is visible | pass/fix | add source labels and credits |
| Prompt has variables | pass/fix | add replaceable fields and criteria |
| Thai is easy to read | pass/fix | shorten paragraphs and split pages |
| Reusable asset is ready | pass/fix | add SOP/checklist/template |

## Playbook Blueprint

Use as an internal planning page before final writing:

```text
Promise:
Problem This Playbook Solves:
What You Can Do After Reading:
Start Here:
Before / After:
Hard Words:
Simplest Example:
Quick Win:
Method:
Page Plan:
Step Pages:
Full Worked Example:
Copy Box:
Fix Guide:
Decision Guide:
Cheat Sheet:
Companion Assets:
```

The blueprint should be decision-complete enough that another agent can write the playbook without inventing the structure.

## CTA Page

Inside the playbook, CTA should tell readers how to use the current file now:

- Start with the Quick Win.
- Fill the worksheet.
- Use the copy box.
- Run the quality check.
- Turn the result into a checklist or SOP.

Do not imply a fuller version exists elsewhere.
