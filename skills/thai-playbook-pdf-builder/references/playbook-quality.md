# Playbook Quality Reference

Load this when deciding structure, reviewing quality, or turning a draft into a publish-ready Claude Work TH Simple Action Playbook.

## What A Good Playbook Does

A good playbook changes the reader's behavior. It gives them a clear output to create, a workflow to follow, and a way to judge whether the result is good enough.

Minimum standard:

- Promise tells the reader what they can build, decide, improve, or reuse after finishing.
- First 5 pages make the problem, outcome, starting path, before/after, and beginner explanation obvious.
- Hard words are explained in plain Thai before advanced steps.
- The simplest example appears before the full method.
- The method has 3-5 named steps and appears throughout the file.
- Every 1-2 pages include an example, template, checklist, worksheet, prompt, table, fix, or action step.
- The playbook includes one full worked example from input to final output.
- The playbook includes a copy box readers can reuse.
- Source layers are separated in planning files and source credit.
- The reader can start within 15 minutes without reading the entire PDF.
- A full package includes `source_audit.md`, `research_notes.md` when needed, and `playbook_blueprint.md` before final PDF work.

## Source Audit Gate

Create this before writing the playbook. It should answer:

- What is the Big Idea from the source?
- Which concepts, workflows, examples, warnings, and limitations are actually in the source?
- Which screenshots or timestamps are useful as visual references?
- Which claims need external verification?
- Which metadata fields are missing?
- What should not be copied directly?

If the audit does not identify a practical workflow, the playbook is not ready to draft.

## Research Gate

Use external research only where it improves execution or verifies unstable facts.

Research should:

- prefer official docs and primary sources
- record URL, access date, usage, and confidence
- separate verification from recommendation
- mark uncertain claims as `needs verification`

Do not add research just to make the file look longer.

## Blueprint Gate

Create `playbook_blueprint.md` before writing the final playbook.

The blueprint must lock:

- sharp promise
- Problem This Playbook Solves
- What You Can Do After Reading
- Start Here
- Before / After
- Hard Words In Plain Thai
- Simplest Example First
- What To Prepare Before Starting
- Quick Win
- 3-5 step method
- page plan for 20-32 pages, or justified 33-40 pages
- step page list
- full worked example
- copy box
- fix guide
- decision guide
- good output vs weak output
- cheat sheet
- companion asset plan

## Problem This Playbook Solves

Use this near the beginning. It should answer:

```text
This playbook helps you stop [pain/problem]
and create [specific output]
using [simple method/source idea].
```

Weak versions use broad words like "learn more" or "understand better" without a visible output. Rewrite them until the reader knows what they will be able to do.

## What You Can Do After Reading

Use this near the beginning:

```text
After reading this playbook, you can:
- create [specific output]
- use [template/checklist/prompt]
- check quality with [criteria]
- apply it to [real work context]
```

## Start Here

Give readers a fast route and a deeper route:

| Situation | Best Path | What To Do |
|---|---|---|
| Need a fast win | Quick Win | do the 15-minute action |
| New to the topic | Beginner path | read hard words and simplest example first |
| Real project | Step-by-step path | follow all step pages and examples |
| Team/process need | SOP path | use decision guide, cheat sheet, and checklist |

## Page-Level Engagement Rules

- Every 1-2 pages must contain a practical reader aid.
- Avoid explanation-only pages unless they establish the foundation, Big Idea, or source context.
- One page should answer one main question.
- Use concrete Thai work scenarios: creator, marketer, owner, operator, team lead, AI workflow user.
- Prefer plain action labels: `Try This`, `Copy This`, `Check Before Use`, `Fix It`, `Next Action`.

Reader aids include:

- mini example
- before/after
- table or matrix
- checklist
- worksheet
- prompt box
- decision guide
- common mistake and fix
- quality check
- 15-minute action

## Good vs Weak Output

Every Simple Action Playbook needs at least one comparison:

```text
Weak Output:
[too vague, generic, unsafe, unsourced, or not usable]

Why It Fails:
[1-3 reasons]

Good Output:
[specific, useful, sourced, and actionable]

Fix:
[steps to turn weak output into good output]
```

Use examples that match the reader's real job, not generic placeholders.

## Quality Check

Use a pass/fix checklist before the final action plan:

```text
Big Idea clear enough: pass / fix
Reader can act in 15 minutes: pass / fix
Hard words are clear: pass / fix
Simplest example is understandable: pass / fix
Inputs are realistic: pass / fix
Source and research labels are clear: pass / fix
Prompt/template has variables and output criteria: pass / fix
Weak output can be improved with the fix steps: pass / fix
Reusable asset is ready: pass / fix
Manual review point is visible: pass / fix
```

Publish-ready standard: any "fix" item must include the exact next edit before readers use it.

## Full Worked Example

Include one example that shows the complete workflow:

1. Situation
2. Input
3. Method step-by-step
4. Draft output
5. Weak point or risk
6. Fix
7. Final output
8. Quality check result

This is often the most valuable section of the playbook. Do not replace it with generic advice.

## Copy Box

Give copy-ready assets readers can reuse:

- prompt
- checklist
- SOP steps
- worksheet fields
- caption or message template
- QA rubric
- handoff note

Every copy box should include variables to replace and a short note on when to use it.

## Simple Action PDF Structure

Default target: 20-32 pages. Use 33-40 only when justified.

1. Cover: title, promise, method preview, same palette as inside pages.
2. Problem This Playbook Solves.
3. What You Can Do After Reading.
4. Start Here.
5. Before / After.
6. Hard Words In Plain Thai.
7. Simplest Example First.
8. What To Prepare Before Starting.
9. Quick Win.
10. The 5-Step Method, or another simple 3-5 step method.
11. Step 1: objective, inputs, steps, checklist.
12. Step 1: example, prompt, or worksheet.
13. Step 2.
14. Step 2 example.
15. Step 3.
16. Step 3 example.
17. Optional Step 4 or implementation guide.
18. Good output vs weak output.
19. Full worked example.
20. Copy box.
21. Fix guide.
22. Decision guide.
23. Common mistakes.
24. Cheat sheet.
25. 7-day action plan.
26. Source credit and research notes summary.

For longer topics, expand step pages and the worked example. Do not add filler theory pages.

## Research and Source Rules

- Use external research for current tools, model behavior, pricing, APIs, platform rules, legal/compliance context, and best practices.
- Prefer official documentation and primary sources.
- Keep external additions practical: verify a claim, fill a missing step, or add execution context.
- Maintain `source_credit.md`.
- Maintain `research_notes.md` when outside sources materially shape the playbook.
- Do not present unchecked claims as facts. Use `needs verification`.

## Visual Rules

- Keep the cover in the same cream/ivory visual family as the inside pages.
- Do not paste screenshots, thumbnails, logos, or creator-owned visuals unless the user has rights and explicitly asks.
- Use source images only to understand the content, then redraw the idea.
- Prefer diagrams, flow cards, mock tables, prompt boxes, worksheets, and decision guides.
- Keep CTA pages focused on using the current complete playbook, not sending readers elsewhere for a fuller version.
- Avoid dense pages. Split pages with more than one main idea.

## Review Questions

- What can the reader do immediately after reading?
- Does the first page feel like the same PDF, not a detached dark cover?
- Do the first 5 pages help beginners understand the point before action?
- Is there a simple repeatable method, not just a list of tips?
- Does each step include an example, prompt/template, or checklist?
- Is there one full worked example?
- Are source material, external research, and Claude Work TH application separated clearly in planning and credit files?
- Are limitations and source credits visible?
- Would the page still be readable as a phone screenshot?
